# 章亦春：把 Nginx 变成可编程平台，用开源改写中国互联网基础设施的男人

> **"机器编码深深植根于 OpenResty 的所有解决方案中。人类只需要向机器输入行业知识和意图，机器就能自己编写、优化及测试代码。"**
> — 章亦春（agentzh），OpenResty 创始人

章亦春可能是中国互联网圈里**最低调的世界级程序员**之一：

- 他在中国雅虎时期创立了 OpenResty 的雏形，后来用 **Nginx + LuaJIT** 把它重写成高性能 Web 平台
- 这个项目如今支撑着全球超过 **4000 万域名**，每天处理**万亿级请求**
- 他是 **Cloudflare 早期核心工程师**，把 OpenResty 推向全球 CDN 的技术标准
- 他前后向 **Linux 内核、Nginx、LuaJIT、GDB、SystemTap、LLVM、Perl** 等核心项目贡献过**超过百万行代码**
- 他写过 **60 多个开源软件库**，包括大量动态追踪和系统诊断工具
- 他倡导的"抄书法"、动态追踪方法论、开源商业化路径，影响了一代中国后端工程师

我们研究他，不是因为他做出了一个流行框架，而是因为他展示了**一条中国技术人极少走通的路径**：从国内大厂出发，用全球开源社区作为杠杆，最终把个人项目变成跨国基础设施，并尝试用"机器编程"这个更宏大的愿景继续推进。

**但请注意**：章亦春的路径依赖特定技术窗口（2007 年 Nginx 崛起、2012 年 Cloudflare 扩张），也经历过阿里 3.25 绩效、开源社区"看门太紧"、商业化弯路等真实代价。我们提炼的是**开源、系统软件、国际化协作的可复制方法**，而不是简单模仿他的履历。

---

## 为什么值得学

对 3-7 年技术人而言，章亦春的独特价值在于：

1. **他证明了"把现有系统变成可编程平台"比"重写一个系统"更有杠杆效应**——OpenResty 没有重新发明 Web 服务器，而是在 Nginx 事件驱动架构上叠加 LuaJIT，让 Nginx 从静态代理变成动态应用平台。这种"站在巨人肩膀上扩展能力边界"的思路，对今天想进入基础设施领域的工程师极有启发。
2. **他展示了从中国大厂走向全球开源基础设施的完整路径**——从雅虎中国到淘宝，再到 Cloudflare，最后创办 OpenResty Inc.，他的每一步都不是简单的跳槽，而是把个人项目不断推向更大场景。
3. **他把"动态追踪"从调试技术上升为系统认知方法**——通过 SystemTap、SystemTap toolkit、OpenResty XRay 等工具，他把"生产系统本身当作可直接查询的数据库"，这套方法论在可观测性时代越来越重要。
4. **他的"抄书法"是一种可立即落地的深度学习技术**——不是泛泛阅读，而是手抄源码、手抄书籍，用"延缓阅读速度"换取深度内化。这个方法对任何想突破"看完就忘"瓶颈的人都适用。

---

## 关键转折

### 转折一：从抄书到教机器解题——深度学习与机器编程的双重起点

章亦春从小专注、好奇心极强。初中时痴迷化学，曾深夜在家做实验引发爆炸；后来转向计算机，家人反而松了一口气。他自述"立志找到值得投入一生的事业，然后就但行好事，莫问前程"。

他的学习方法非常特别：**抄书**。高中时借到一本大学《分析化学》教材，他花了整个暑假手抄全书。后来进入淘宝想学 Nginx 源码，他白天对着 Kindle 手抄 C 源代码，晚上在房间里踱步，在脑海中反复回放白天抄过的代码，直到融会贯通。

他后来解释："抄书的奥妙在于延缓阅读速度，不至遗漏每一个重要细节：眼到，手到，心到。"

**关键信号**：章亦春的深度学习不是天赋，而是**可复制的刻意练习方法**。他后来的机器编程愿景，也源自高中时代"教机器解高考题"的尝试——他一直在思考：如何把人类意图高效转化为机器实现。

---

### 转折二：加入中国雅虎搜索技术部（2007）——OpenResty 的萌芽

2007 年大学毕业后，章亦春加入中国雅虎搜索技术部（4E 团队）。这个机会来自他在毕业前夕受邀在 PerlChina 北京大会上做开源分享，正好在中国雅虎总部认识了这个团队。

当时正值 OpenAPI 热潮兴起，团队需要一套抽象的 Web Service 平台。在公司领导支持下，章亦春从早年开始做 OpenResty 的雏形，并且**从一开始就开源**。第一代 OpenResty 用 Perl 和 Haskell 实现，定位是"meta web service"：让用户基于 REST 风格组合出新的 Web Service。

**关键信号**：OpenResty 的起点不是"我要做一个伟大项目"，而是**解决一个真实的内部需求，并且选择开源来获得外部反馈**。这个"问题驱动 + 开源放大"的模式，贯穿了他的整个职业生涯。

---

### 转折三：阿里 3.25 绩效与淘宝数据部门重写 OpenResty（2009）——从挫折到聚焦

2008-2009 年，阿里巴巴对中国雅虎进行重组拆分。章亦春被分配到淘宝搜索部门，但那段经历对他来说并不愉快：开很多会、经常被拉到杭州出差、OpenResty 开发停滞。他一度想辞职。

关键时刻，淘宝商户数据统计部门的老大姜蕾和张茂森找到他，邀请他加入数据部门。双方达成了一个对彼此都有利的协议：章亦春可以继续做开源 OpenResty，而淘宝数据平台将基于 OpenResty 建设。

在此期间，他与同事王晓哲（网络 ID：chaoslawful）一起，**基于 Nginx 和 LuaJIT 彻底重写了 OpenResty**。原来的 Perl/Haskell 版本被废弃，新版 OpenResty 把 Lua 脚本语言嵌入 Nginx，让 Nginx 获得了强大的动态编程能力。

两年后，整个淘宝数据分析平台迁移到新版 OpenResty 上，实现了性能数量级的提升。

**关键信号**：这次转折展示了章亦春的**两个核心能力**：
- **在组织约束中寻找空间**：他没有硬碰硬地对抗不喜欢的环境，而是找到愿意支持他的团队
- **敢于彻底重写**：第一代 OpenResty 已经有一定基础，但他不恋旧，为了性能和生态选择推倒重来

值得注意的是，章亦春早期在阿里曾因"价值观"问题拿到 3.25 的不合格绩效。这件事提醒我们：即使是世界级工程师，也可能在大厂评价体系里遭遇误判。

---

### 转折四：福州"田园生活"全职开源（2011）——主动选择的清贫与自由

2011 年，章亦春觉得在淘宝的使命已经完成，如果再往下走会进入"开会 PK 项目"的阶段，这不符合他的性格。他再次辞职，和当时的女朋友（后来的夫人）一起搬到一个南方小城福州，过了一年"田园生活"，在家全心投入 OpenResty 等开源项目。

这段时间他没有任何正式收入，生活靠积蓄支撑，但获得了最宝贵的东西：**完整的时间自主**。

**关键信号**：章亦春的"田园生活"不是逃避，而是**为长期目标购买时间**。对技术人来说，真正的稀缺资源往往不是钱，而是能连续数月不受打扰地钻研一个硬核问题的机会。

---

### 转折五：加入 Cloudflare（2012）——从中国开源走向世界基础设施

2011 年 6 月，Cloudflare 创始人 Matthew Prince 和 Michelle Zatlyn 第一次邀请章亦春加入。那时章亦春正准备去福州，婉拒了邀请。Matthew 极其有诚意，出资邀请他们去旧金山体验一周。章亦春体验后仍然选择回福州。

但 Matthew 没有放弃。2011 年 11 月他到北京再次邀请章亦春，最终被他的诚意打动。2012 年 6 月，章亦春和夫人正式搬到旧金山湾区。

在 Cloudflare 期间，章亦春全职从事 OpenResty 开源开发，以及基于 OpenResty 的全球 CDN 网络软件系统研发。Cloudflare 的大量边缘节点成为 OpenResty 的实战熔炉，OpenResty 也因此在性能、稳定性和生态上突飞猛进。

**关键信号**：章亦春加入 Cloudflare 不是为了一份美国工作，而是为 OpenResty 找到了**全球最大的分布式测试床和商业化前哨**。这提醒我们：选择平台时，要看它能否把你的项目推向更真实的极端场景。

---

### 转折六：创办 OpenResty Inc.（2017）——从开源项目到商业化与机器编程

2016 年底，章亦春离开 Cloudflare。2017 年初，他在美国创办 OpenResty Inc.。

这个决定有两个驱动：
1. **个人愿景**：他希望在 OpenResty 上实现更宏大的"机器编程"蓝图，而不是受雇于企业时被业务需求分散精力
2. **社区局限**：OpenResty 渗透到越来越多公司和行业，但开源社区无力提供商业级别的技术支持

公司推出了 **OpenResty XRay**（实时分析与诊断平台）和 **OpenResty Edge** 等产品，试图把动态追踪、可观测性和机器编程结合起来。章亦春坦言，公司前五年走了不少弯路：过度低调、商业化投入不足、团队清一色工程师。

**关键信号**：章亦春从工程师到 CEO 的转变，是很多开源项目创始人都会面临的挑战：**技术影响力可以自然增长，商业化和组织建设却需要主动学习**。

## 核心心法与习惯

### 1. 抄书法：用"慢"换"深"

章亦春逢人就推荐"抄书"：

> "抄书的奥妙在于延缓阅读速度，不至遗漏每一个重要的细节：眼到，手到，心到。"

他在淘宝时学 Nginx 源码，就是白天手抄 C 代码，晚上在脑海中回放。一两个月后，他觉得自己"大彻大悟"。

**可学习的**：
- 不要只读源码，要**抄写关键路径**的源码
- 不要只抄一遍，要**在不同时间段反复回放**
- 选择真正经典的代码，而不是流行框架的表面 API

---

### 2. 把生产系统当作"可直接查询的数据库"

章亦春是动态追踪技术的重度实践者。他写了很多 SystemTap 工具，包括 nginx-systemtap-toolkit、perl-systemtap-toolkit、stappxx 等。

他的方法论是：
- 不要在生产系统里到处埋点打日志
- 而是把整个软件栈看成可以直接查询的数据库
- 在关键"穴位"放置探针，采集所需信息，用完即走，不留痕迹

他推崇"小步推进、连续求问"：把大问题拆成一系列小假设，用简单工具逐步验证，而不是写一个很复杂的多用途工具。

**可学习的**：
- 学习一门动态追踪工具（SystemTap、eBPF、DTrace、OpenResty XRay）
- 遇到线上疑难问题时，先问"能不能不改代码就定位"
- 复杂调试工具本身容易引入 bug，优先用简单、针对性强的工具

---

### 3. 开源商业化要做"互补价值"，不要做"替代价值"

章亦春对开源商业化有一个关键判断：

> "商业化的产品和开源产品不能在同一个或者类似的赛道上，尤其是在开源产品已经很强大的情况下。开源商业化要取得成功，关键就是要提供与开源项目互补（而不是替代之）的价值，让市场愿意为这个价值买单。"

这也是 OpenResty XRay 的定位：开源 OpenResty 已经很强，但企业需要商业级的实时诊断、性能分析和技术支持，这就是互补价值。

**可学习的**：
- 如果你在做开源项目，想清楚开源版本解决什么、商业版本解决什么
- 商业版本最好是"开源用户也会头疼的问题"，而不是"开源功能的闭源版"
- 不要低估迁移成本、内部整合等企业采购中的非技术因素

---

### 4. "稳定"与"开放"的平衡

章亦春认为 OpenResty 前十五年的关键词是"稳定"，未来十五年的关键词应该是"开放"。

他承认过去自己"看门太紧"，对贡献者的补丁审查非常严格，导致社区贡献者数量远低于用户数量。现在他在调整：建立 Slack workspace、引入 Canny 需求投票、使用 Aviyel 贡献者激励平台、组织更多 meetup。

**可学习的**：
- 项目早期需要严格把控质量来建立信任
- 但长期来看，**过度严格会扼杀社区活力**
- 质量与开放之间需要动态平衡

---

### 5. 机器编程：把人类从重复开发中解放出来

章亦春对 OpenResty 的终极愿景是"机器编程"：

> "人类只需要向机器输入行业知识和意图，机器就能自己编写高效可靠的实现代码、自行测试、快速部署在大规模计算机系统上，并同时生成人类可读的文档。"

这个愿景源自他高中时教机器解高考题的经历。他认为，未来程序员应该像现代制造业工人受益于机器人自动化一样，从繁琐的商业逻辑和重复性开发中解放出来，专注于创新型工作。

**可学习的**：
- 关注 AI 辅助编程、代码生成、自动化测试、自动运维等方向
- 但不要神化"机器编程"——它目前还处在早期探索阶段
- 真正的价值在于：把重复性工作交给机器，把创造力留给人

## 可复制的行动（普通人今天就能开始）

### 🔵 今天可以开始的 3 件事

1. **用"抄书法"读一个你每天都在用的系统软件的源码**
   - 选择 Nginx、Redis、Linux 内核某个子系统、或你项目依赖的核心库
   - 不要贪多，先选一条关键路径（比如 Nginx 的 HTTP 请求处理流程）
   - 手抄或逐行注释 500-1000 行代码，晚上在脑中复盘

2. **给你使用的一个开源项目提交一个真实反馈**
   - 可以是一个 bug 报告、一段文档改进、一个测试用例
   - 关键是迈出与上游社区协作的第一步
   - 推荐从 OpenResty、Nginx、Redis、Linux 等你实际依赖的项目开始

3. **学习一种动态追踪或诊断工具**
   - 入门可以选择 `perf`、`bpftrace`、`OpenResty XRay`（试用版）或你公司已有的 APM 工具
   - 目标是：遇到线上问题时，能先不改代码就定位根因
   - 记录一次你用它解决的真实问题

### 🟡 本季度可以做的事

- **建立你的"可编程平台"思维**：选一个你工作中重复出现的场景，问自己：能不能在现有基础设施上增加一层脚本/插件/扩展能力，而不是重写整个系统？
- **参与一次真实的技术社区协作**：邮件列表、GitHub Issue、Slack/Discord 讨论都可以。重点是学会"给别人有效反馈"和"接受别人反馈"。
- **写一份"系统动态追踪"实践笔记**：记录你用动态追踪或日志分析定位线上问题的完整过程，发布到你的博客或团队 Wiki。

## 局限与代价

### 1. 严格审查导致社区贡献门槛过高

章亦春对代码质量把控极严，这保证了 OpenResty 的稳定性，但也让很多潜在贡献者望而却步。他自己也承认："看门太紧对项目的可持续发展性造成了一些影响。"

**启示**：质量是开源项目的生命线，但**过度严格会让社区变成一言堂**。找到适合你项目阶段的审查平衡点。

### 2. 机器编程愿景宏大但实现周期长

"机器编程"是一个非常有吸引力的愿景，但距离真正大规模落地还有很长的路。OpenResty Inc. 前五年主要在做技术积累，商业化成果相对有限。

**启示**：宏大愿景可以指引方向，但团队也需要能在中短期内产生现金流和正向反馈的产品。

### 3. 开源成功不等于商业成功

OpenResty 作为全球使用量极大的开源项目，其商业化公司 OpenResty Inc. 的市场知名度却远低于项目本身。章亦春自己也承认"之前的低调导致成果不为人所知"，且早期团队"清一色工程师，没有商业人才"。

**启示**：技术影响力是商业化的必要条件，但不是充分条件。**商业化需要专门的人才、投入和方法论**。

### 4. 个人风格与组织扩张的张力

章亦春是极度内向、喜欢专注技术的人。作为 CEO，他不得不把大量时间花在管理、客户交流和公司事务上。这种角色转换对很多技术型创始人都是巨大挑战。

**启示**：如果你决定从开源项目创始人转型为创业者，要评估自己是否愿意花大量时间在非技术事务上，或者是否能找到互补的合伙人。

## 代表作品

- **OpenResty**：基于 Nginx 和 LuaJIT 的高性能 Web 平台，支撑全球 4000 万+ 域名，日处理万亿级请求
- **ngx_lua / lua-nginx-module**：让 Lua 脚本在 Nginx 各个处理阶段运行的核心模块
- **lua-resty-* 系列库**：OpenResty 生态中的 MySQL、Redis、Memcached、WebSocket 等客户端库
- **nginx-systemtap-toolkit / perl-systemtap-toolkit / stappxx**：生产环境动态追踪工具集
- **OpenResty XRay**：实时分析与诊断平台，无需修改应用即可定位性能、异常和安全问题
- **OpenResty Edge**：基于 OpenResty 的分布式流量管理和边缘计算平台

# dana · 大拿

> **研究大拿 · 成为大拿**

**dana** 是一个**严肃的开源知识平台**，致力于系统研究技术大拿的成长路径、心法与决策，并帮助有志者通过可复制的方式成为更好的自己。

我们相信：**大拿不是天生的，而是长期主义 + 正确方法 + 批判性学习 + 持续输出的结果。**

---

## 项目使命

- **研究大拿**：用严谨的方法论，深度剖析真正优秀的工程师
- **成为大拿**：把研究成果转化为可落地的成长路径和心法

我们拒绝鸡汤、偶像化、成功学，追求**高质量、可复制、批判性**。

> [!important] 当前状态
> 知识库已进入成熟阶段（40+ 篇高质量内容 + 研究方法论 + 系统路径 + 可视化 + AI 时代完整小专栏）。欢迎直接使用与贡献。本轮 1 2 3 完成 3 位 AI 大拿深化 + 2 新案例 + 1 心法实战案例扩充。

## 为什么存在这个库？

市面上有太多「XXX 技术从入门到精通」，却很少有人系统回答：

- 一个普通工程师，怎样在 5-10 年内真正脱颖而出？
- 那些我们仰望的「大拿」，他们的关键转折点和心法到底是什么？哪些是可以学习的？
- 在信息爆炸的时代，如何建立属于自己的技术判断力与影响力？

dana 试图用**研究方法论 + 系统路径 + 高质量人物复盘**，给出一个严肃、可操作的回答。

---

## 这个知识库适合谁？

- 工作 2-8 年的技术人，感到「天花板」或「瓶颈」，想系统性突破
- 希望从「写好业务代码」进化到「有技术影响力」的人
- 愿意向优秀榜样学习，但不想盲目崇拜，想要**可复制经验**的人
- 正在构建个人知识体系、第二大脑的技术人

**不适合**：只想快速拿 offer 面试突击、纯技术工具速成、寻求鸡汤的人。

## 如何参与研究大拿

我们欢迎任何人参与“研究大拿”：

1. **提出研究建议**：在 GitHub Issues 中提交你认为值得深入研究的大拿
2. **提交研究成果**：按照 [[研究大拿的方法论]] 研究后，提交 PR
3. **分享真实案例**：匿名分享你自己的成长复盘（尤其是 3-7 年突破故事）
4. **改进方法论**：对研究框架提出建议

所有高质量贡献都会被认真 review 并纳入知识库。

## 如何使用这个知识库？

### 推荐阅读路径

1. **先读索引**：从 `索引/` 开始，快速建立全貌
2. **人物志先行**：挑 2-3 位最打动你的人物，精读他们的「可学习点」
3. **心法内化**：把 `心法与原则/` 里的模型抄到自己的笔记或卡片里
4. **对标路径**：找到自己当前阶段的文章，制定 3-6 个月小目标
5. **持续输出**：每读一篇，写一段「我能立刻开始的 1 件事」

### Obsidian / Logseq 使用建议

- 直接把整个仓库作为 vault 打开
- 充分利用 `[[wikilinks]]`（Obsidian 双链语法）和标签系统
- 推荐建立自己的 `我的复盘/` 文件夹，把读后感和真实案例写进去
- 定期 review 索引文件，保持知识库鲜活

## 内容原则（我们的质量承诺）

- **绝不鸡汤**：每一条原则都配真实故事、具体案例或可验证方法
- **可复制优先**：讲大拿，更讲「普通人今天能抄的作业」
- **批判性学习**：偶像也会有局限，我们提炼的是「心法」而非「人设」
- **长期主义**：不追求短期速成，强调 3-10 年的复利
- **中文友好**：以中文思考和表达为主，同时保留全球视野

## 首批人物选择标准

我们挑选人物的标准是：

1. **可学习性高**：其成长路径有清晰、可复制的阶段和方法
2. **影响力真实**：在技术社区或公司内被公认的「大拿」而非营销
3. **多元覆盖**：既有大厂技术专家，也有独立开发者、教育者、极致深度专家
4. **公开资料丰富**：有足够公开的访谈、文章、演讲可供提炼

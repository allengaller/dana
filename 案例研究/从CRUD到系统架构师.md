# 从 CRUD 到系统架构师：一个后端工程师的 3 年突破路径

> 「我不是天才，我只是在别人刷短视频的时候，把 Redis 源码读了三遍。」

---

## 背景

**人物**：化名「陆远」，男，某二线城市互联网公司后端工程师。

**阶段**：工作第 3 年，日常写 Java 业务 CRUD，熟练使用 Spring Boot + MyBatis，能独立完成中等复杂度的业务模块。

**领域**：电商后台系统，团队约 40 人，后端 12 人。

**约束条件**：

- 非 985/211 本科，无大厂经历
- 所在城市技术社区活跃度低，身边缺少高水平同行
- 工作内容以业务需求交付为主，每天能自由支配的学习时间约 1.5 小时
- 团队里没有真正的架构师带路，技术决策基本靠 leader 经验

**当时的困境**：陆远发现自己和周围同龄人的差距在缩小——大家写的代码都差不多，做的需求都差不多。他开始焦虑：如果继续这样下去，5 年后自己只是一个「写了 8 年 CRUD 的高级工程师」，而不是真正的技术专家。这正是 [[3-7年：垂直深挖与第一次突破]] 描述的典型瓶颈。

---

## 关键决策

### 决策一：选定「分布式系统」作为垂直深挖方向

陆远花了两周时间分析自己公司和合作公司的技术需求，发现一个共同痛点：**几乎所有中等规模团队都在从单体向分布式过渡的过程中踩了大量坑**，但真正理解分布式原理、能设计合理方案的人极其稀缺。

他判断：这个方向既有真实需求（不是伪需求），又有足够深度（可以持续投入 3-5 年），而且能直接应用到自己的工作中。

这个选择体现了 [[T型深 + 梳型广]] 的核心思路——先在一个有真实需求的子领域打透，而不是什么都浅尝辄止。

### 决策二：用「源码阅读 + 造轮子」代替纯看书学习

他之前也买过不少书，但发现看完就忘，停留在「知道但不会用」的状态。受 [[源码阅读与造轮子策略]] 启发，他决定改变学习模式：**以读源码为主、写轮子为练、看书为辅**。

### 决策三：主动在公司内寻找「超出当前能力」的项目

他找 leader 谈了一次，主动请求参与公司正在规划的缓存中间件升级项目——一个当时他还不完全有把握的项目。这是典型的 [[做难事：突破舒适区]]。

---

## 行动过程

### 第一阶段：源码阅读打地基（第 1-6 个月）

陆远制定了严格的 [[刻意练习与及时反馈]] 计划，每天固定 1.5 小时用于源码阅读，持续 6 个月：

**阅读清单与节奏**：

| 月份 | 项目 | 目标 | 产出 |
|------|------|------|------|
| 1-2 | Redis（重点读网络模型和数据结构） | 理解单线程为什么快、事件驱动怎么设计 | 3 篇源码分析笔记（每篇 3000 字以上） |
| 3-4 | RocketMQ（重点读消息存储和消费模型） | 理解高吞吐写入和顺序消费的底层实现 | 2 篇源码分析 + 1 张完整架构流程图 |
| 5-6 | etcd / Raft 论文实现 | 理解分布式一致性在实际系统中的落地 | 1 篇深度笔记 + 1 次团队内部分享 |

**关键方法**：不是从头到尾读代码，而是**带着问题读**。每次读之前先写下 3 个具体问题（例如「Redis 的 pipeline 是怎么减少网络往返的？」），读完后用代码注释 + 流程图回答。

每读完一个模块，他就用 [[复盘三问法]] 做一次总结：我原来以为是怎么工作的？实际上是怎么工作的？我的认知偏差在哪里？

### 第二阶段：造轮子强化理解（第 7-12 个月）

读完源码之后，他进入「造轮子」阶段。这不是为了替代成熟方案，而是为了**在写代码的过程中暴露自己的理解盲区**：

1. **用 Go 实现了一个简化版的分布式 KV 存储**（约 5000 行代码）
   - 包含：Raft 选举、日志复制、简单分片
   - 耗时：利用下班和周末，约 3 个月
   - 最大收获：真正理解了「网络分区时系统会怎么表现」——这是看 100 遍论文都不会有的体感

2. **给公司的 Redis 集群写了一个轻量级的连接池和监控中间件**
   - 这个轮子直接解决了公司当时 Redis 连接泄漏的问题
   - 被 leader 认可，正式纳入公司内部工具库

这个阶段是 [[项目驱动学习法]] 的典型实践——他造的轮子不是玩具，而是有真实使用场景的东西。真实用户（同事们）的反馈成了最高质量的反馈回路。

### 第三阶段：在真实项目中验证和突破（第 13-24 个月）

陆远开始主动承接公司内与分布式相关的「难事」：

- **主动认领了订单系统的分库分表改造项目**：这是团队里没人想碰的「脏活」，涉及历史数据迁移、双写一致性、灰度切换等复杂问题。他花了 4 个月主导完成，期间经历了 2 次方案推翻重来。
- **设计并落地了公司的统一配置中心**：参考 etcd 的设计思路，从 0 到 1 搭建，支撑了全公司 30+ 微服务的配置管理。
- **推动了全链路压测体系的建立**：他画出了完整的业务链路图（实践 [[系统思维与全链路视角]]），识别出 3 个关键瓶颈点，推动了压测平台的采购和落地。

每一次项目结束后，他都会写一篇 2000-4000 字的技术复盘，发到公司内部知识库和掘金平台。这就是 [[输出倒逼输入]] 的实践——写作过程迫使他重新审视每一个设计决策，而公开后的评论和讨论又带来了高质量的反馈。

### 第四阶段：影响力扩展（第 25-36 个月）

- 在掘金上持续输出分布式系统系列文章，累计 20+ 篇，获得约 8000 关注者
- 受邀在 QCon 某分论坛做了一次 30 分钟的分享
- 被一家中型技术公司的 CTO 通过文章联系到，邀请加入担任系统架构师

## 结果与数据

| 维度 | 3 年前 | 3 年后 |
|------|--------|--------|
| 技术深度 | 熟练使用 Spring Boot 做 CRUD | 能从源码层面解释 Redis / MQ / 一致性协议的设计决策 |
| 项目层级 | 负责单个业务模块 | 主导跨团队的架构项目（分库分表、配置中心、全链路压测） |
| 影响力 | 公司内无人知晓 | 掘金 8000 粉、受邀在技术大会分享 |
| 职位 | 普通后端工程师 | 中型公司系统架构师（薪资涨幅约 80%） |
| 可验证产出 | 无 | 2 个开源工具、20+ 篇深度文章、3 个落地架构项目 |

**关键转折点**：第 14 个月，分库分表项目成功上线后，他开始被公司内其他团队主动求助分布式相关问题。这是从「普通工程师」到「被认可的技术专家」的标志性时刻。

## 可复制点

### 1. 选方向的判断框架

不要选「最热门」的方向，选「你的工作环境中真实存在痛点」的方向。陆远选分布式系统，不是因为它时髦，而是因为他观察到公司和合作公司都在这个方向上反复踩坑。**有真实需求的方向，你的学习才能和工作形成正循环。**

### 2. 源码阅读不是「看」，是「带着问题找答案」

很多人读源码效率低，是因为没有明确问题。陆远的方法是每次阅读前写下 3 个具体问题，读完后用笔记回答。这把「被动浏览」变成了「主动探索」，效率提升巨大。参见 [[源码阅读与造轮子策略]]。

### 3. 造轮子的目的不是「替代」，是「理解」

陆远造的 KV 存储从来没有替代过任何生产系统，但写这 5000 行代码的过程，比读 10 本分布式系统的书学到的都多。**写的过程中你会遇到无数「看论文时根本想不到」的工程细节。**

### 4. 主动认领「没人想碰」的项目

分库分表改造是典型的「脏活累活」，但正是这种项目给了他最密集的成长机会。正如 [[做难事：突破舒适区]] 所述——**别人不愿意做的事里，藏着最稀缺的经验值**。

### 5. 输出是最好的学习加速器

如果陆远只是默默读源码、默默造轮子，他的成长速度大概会慢一半。持续写作迫使他「把模糊的体感变成清晰的表述」，而公开发表带来的反馈则不断修正他的认知偏差。参见 [[输出倒逼输入]] 和 [[写作与分享的杠杆]]。

### 6. 时间投入不需要很多，但必须持续

陆远没有辞职闭关学习，也没有每天熬夜到凌晨。他每天只投入 1.5 小时，但坚持了 3 年，累计约 1600+ 小时的刻意练习。**关键不是爆发力，而是持续性。**

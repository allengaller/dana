# 0-3 年：根基与判断力

> 这个阶段的核心任务：**不要急着成为架构师，先成为一个靠谱、能独立交付、懂底层的人**。
> 根基不牢的人，后续每一个阶段都会付出加倍的代价。

---

## 为什么这个阶段如此重要？

0-3 年是技术人职业生涯的「地基期」。这个阶段打下的基础，决定了你后续 10-20 年能走多高、走多远。

很多后来成为大拿的人，回顾这个阶段时都说：**当时觉得在「笨笨地打基础」，后来发现那几年学到的底层知识，让自己在每一个新技术浪潮中都能快速上手**。

而很多在这个阶段「追框架、跳公司、只做 CRUD」的人，到了 5 年、8 年时会遇到一个残酷的现实：经验年限在增长，但能力并没有本质提升。

---

## 这个阶段必须建立的 5 项核心能力

### 1. 写出「经得起 review」的代码

不是「能跑就行」的代码，而是：

- **命名清晰**：变量名、函数名让同事一眼看懂意图
- **结构合理**：一个函数做一件事，模块边界清晰
- **考虑异常**：不只是 happy path，还有边界、超时、失败处理
- **可测试**：核心逻辑有单元测试覆盖

**具体练习**：每次提交代码前，问自己「如果这段代码半年后由一个新人来维护，他能看懂吗？」

### 2. 独立排查中等复杂度的线上问题

从「遇到问题找 leader」到「自己能全链路定位问题」：

- 能看懂日志、监控、调用链
- 能区分是代码 bug、配置问题、依赖服务异常还是基础设施故障
- 能用二分法、排除法快速缩小范围
- 修复后能写清楚 RCA（Root Cause Analysis）

**为什么重要**：线上排查能力是区分「初级」和「靠谱」的最直接标志。一个能独立排查问题的人，会被团队真正信任。

### 3. 至少读懂一个真实生产系统的核心模块

不是看教程里的 demo，而是真正理解一个生产级系统：

- 请求从入口到返回的完整链路
- 数据是如何存储、流转、保证一致性的
- 系统在什么条件下会出问题（并发、网络分区、磁盘满）
- 为什么代码要这样写（而不是你想象的「更优雅」的方式）

**推荐做法**：
- 读公司核心项目源码（从你负责的那个模块开始）
- 用 [[源码阅读与造轮子策略]] 的方法，选一个开源项目深入读
- 画架构图、画数据流图、画时序图——画不出来说明你没真懂

### 4. 开始建立「技术判断力」的第一批直觉

技术判断力不是天生的，而是通过一次次「选择 → 结果 → 反思」循环建立的：

- 什么时候该用缓存？什么时候不该？
- 什么时候该引入新组件？什么时候该用现有方案凑合？
- 什么时候该写「快糙猛」的代码？什么时候必须认真设计？

**建立方法**：
- 每次做技术选型时，写下你的理由
- 3 个月后回头看，哪些判断是对的，哪些是错的
- 用 [[复盘三问法]] 把每一次技术决策变成学习机会

### 5. 形成持续学习的系统习惯

不是「有空就学」，而是把学习变成日常系统的一部分：

- **每天 30 分钟**：读一篇技术文章/论文/源码
- **每周 1 次**：动手做一个小实验（新技术、新工具、新思路）
- **每月 1 次**：写一篇学习笔记或技术总结（参见 [[输出倒逼输入]]）
- **每季度 1 次**：回顾本季度学到了什么，下一步方向是什么

## 0-3 年的 5 个常见误区

### 误区 1：疯狂追新框架，基础不牢

每出一个新框架就想学，简历上写了一堆技术栈，但问到底层原理一个都说不清楚。

**真相**：框架 3-5 年会换一波，但操作系统、网络协议、数据结构、分布式原理 20 年不变。**基础知识的半衰期远长于框架知识**。

### 误区 2：只写 CRUD，从不碰难事

日常工作只做增删改查，遇到复杂问题就绕过去或交给别人。

**真相**：成长只发生在「能力边缘」。如果你 3 年都在做舒适区内的事，你的经验不是 3 年，而是 1 年重复了 3 次。参考 [[刻意练习与及时反馈]]。

### 误区 3：不输出，学了就忘

看了很多文章、听了很多分享，但从来没有自己写过一篇技术总结。

**真相**：不输出的学习，留存率不到 20%。输出是最高效的学习方式，参见 [[输出倒逼输入]]。哪怕只是写给自己看的笔记，也比「只看不写」强 10 倍。

### 误区 4：频繁跳槽，追求 title 而非能力

每 1-2 年跳一次，从初级跳到中级再跳到高级，但实际能力还停留在初级。

**真相**：Title 可以靠跳槽涨，但能力只能靠在一个系统上长期深耕来积累。大拿们几乎都有在某个系统/领域上「啃硬骨头」3 年以上的经历。

### 误区 5：忽视软技能，认为「代码好就行」

不参加 code review，不写文档，不跟同事沟通技术方案。

**真相**：技术能力只是基础，能把技术能力「传递出去」让别人受益，才是真正的影响力起点。这个阶段虽然不需要大影响力，但至少要做到：**能清楚地在技术讨论中表达自己的观点**。

## 0-3 年可以做的 10 件高价值事情

1. **精读一个开源项目的核心模块**（Redis、Nginx、Spring 等，不求多，求深）
2. **独立排查并解决至少 3 个线上问题**（写 RCA 文档）
3. **写 20 篇以上技术笔记**（不求公开，但求写清楚）
4. **参与至少 50 次 code review**（包括给别人 review 和被 review）
5. **完整理解一个生产系统的架构**（画出全链路图）
6. **学会使用至少一个性能分析工具**（profiler、火焰图、慢查询分析）
7. **理解数据库索引原理**（B+ 树、事务隔离级别、执行计划）
8. **理解网络基础**（TCP/IP、HTTP/HTTPS、DNS、负载均衡）
9. **用一门新语言写一个小项目**（拓宽思维，理解不同范式）
10. **找到 1-2 个 mentor**（公司内或社区里比你强 2-3 级的人）

## 这个阶段如何判断自己「准备好了」？

当你能够同时做到以下几点时，说明你已经为 [[3-7年：垂直深挖与第一次突破]] 做好了准备：

- [ ] 你能独立设计并交付一个中等复杂度的系统（不只是写 CRUD）
- [ ] 你能在 code review 中发现别人发现不了的设计问题
- [ ] 你能向非技术人员解释清楚一个技术方案的 trade-off
- [ ] 你开始对某个子领域产生真正的兴趣和直觉
- [ ] 你有了 20+ 篇技术笔记或博客

## 真实案例参考

**案例 A**：某后端工程师前 3 年专注在公司的交易系统上，把每一行核心代码都读透了。到第 3 年时，他已经能独立定位任何线上问题，并开始提出架构优化建议。后来他成为该系统的首席架构师。（参见 [[从CRUD到系统架构师]]）

**案例 B**：某前端工程师前 3 年不只是写业务代码，还深入读了 React 和 Vue 的源码，理解了虚拟 DOM 和响应式原理。到第 3 年时，他能为团队做技术选型，并开始在公司内分享框架原理。（参见 [[源码阅读与造轮子策略]]）

# 源码阅读与造轮子策略：技术人深度成长的两条主路

> 想真正变强，有两条最有效的路：
> **读别人写得好的代码**，和**自己写一个轮子**。
> 两条路交替走，比任何培训课程都有效。

---

## 为什么读源码是技术人的必修课？

大多数工程师的日常工作是「写业务代码」——调用框架、拼装 API、处理业务逻辑。这本身没有问题，但长期只做这件事会导致一个严重的后果：**你对底层原理的理解停留在「听说过」的层面，而不是「真正理解」**。

读源码的价值在于：

### 1. 看到「好代码」长什么样

你在公司里看到的代码，大概率是被业务压力、历史包袱、多人协作扭曲过的。而优秀的开源项目——尤其是那些经过 [[John Carmack]]、[[Rob Pike]] 这个级别工程师打磨的项目——展示的是**工程美学的上限**。

### 2. 理解设计决策的 Why，而不只是 What

文档告诉你「这个 API 怎么用」，源码告诉你「为什么这么设计」。比如：

- 为什么 Go 的 `net/http` 包要那样设计中间件链？
- 为什么 Redis 选择单线程模型而不是多线程？
- 为什么 Linux 内核的调度器要用 CFS？

这些问题的答案，只在源码里。

### 3. 培养「系统品味」

[[章亦春]]（OpenResty 作者）曾说过，读好代码就像听好音乐——你会逐渐建立起一种直觉，知道什么是好的设计、什么是 hack、什么是可以接受的 trade-off。这种品味是做架构师的核心能力。

## 源码阅读的方法论

### 方法一：自顶向下（Top-Down）

从「用户视角」出发，带着具体问题读：

1. **选一个你日常使用的功能**（比如 Redis 的 `GET` 命令）
2. **从入口函数开始追踪**（`networking.c` 里的请求处理 -> 命令分发 -> 具体命令实现）
3. **画出调用链图**
4. **重点关注**：数据结构的选择、错误处理方式、并发模型

适合场景：**对一个项目有一定使用经验，想深入理解某个模块**。

### 方法二：自底向上（Bottom-Up）

从「基础设施」出发，理解底层机制：

1. **先看核心数据结构**（比如 Redis 的 SDS、dict、ziplist）
2. **理解内存管理模型**
3. **理解并发/IO 模型**
4. **然后往上看应用层如何使用这些基础设施**

适合场景：**想系统性理解一个项目的架构**。

### 方法三：问题驱动（Problem-Driven）

最高效的读源码方式：

1. **带着一个具体问题**（「Redis 是怎么处理过期 key 的？」）
2. **直接搜索相关代码**
3. **只读和问题相关的部分**
4. **写笔记总结答案**

适合场景：**时间有限，想快速获取特定知识**。这也是符合 [[刻意练习与及时反馈]] 原则的方式——目标明确、有即时反馈（你能回答那个问题了）。

### 阅读时的关键习惯

- **不要试图读懂每一行**。先读 20% 的核心代码，理解 80% 的设计思想
- **画图**。调用关系图、数据流图、时序图——不画图你不可能真正理解
- **写笔记**。每读完一个模块，用 200 字总结它的核心设计
- **跑调试器**。用 `gdb` 或 IDE debugger 单步执行关键路径，观察运行时状态
- **关注注释和设计文档**。好项目的注释往往比代码本身更有信息量

## 值得读的源码推荐（按难度递进）

### 入门级（代码量小、结构清晰）

- **Redis**：约 5 万行 C 代码，结构极其清晰，是公认的源码阅读入门首选
- **SQLite**：设计优雅、文档详尽，适合理解数据库引擎
- **Go 标准库的 `net/http` 和 `sync` 包**：学习工业级 Go 代码的最佳入口

### 中级（有一定复杂度、设计精妙）

- **Go 运行时（runtime）**：理解 goroutine 调度、GC、内存分配
- **Nginx**：理解事件驱动架构和高性能网络编程
- **LevelDB / RocksDB**：理解 LSM-Tree 存储引擎，[[丁奇]] 在 MySQL/InnoDB 方面的深入工作也与此相关

### 高级（系统级、需要深厚基础）

- **Linux 内核**（选择子系统读，如调度器、文件系统、网络栈）
- **PostgreSQL 查询优化器**
- **Chromium 渲染引擎**

### 中文社区优质项目

- **陈硕的 Muduo 网络库**：[[陈硕]] 写的 C++ 高性能网络库，代码量适中，设计教科书级别
- **OpenResty**：[[章亦春]] 基于 Nginx + LuaJIT 构建的高性能 Web 平台

## 造轮子策略：通过「重新发明」来真正理解

### 为什么造轮子如此有效？

[[项目驱动学习法]] 告诉我们，学习最高效的方式是围绕一个具体项目来组织。而「造轮子」是项目驱动学习的最极致形式：

- 你必须理解**问题的本质**（为什么要这个工具？）
- 你必须理解**设计的 trade-off**（为什么别人那样设计？我能不能更好？）
- 你必须处理**边界情况和错误处理**（这比写出 happy path 重要 10 倍）

### 造轮子的正确姿势

**第一原则：先造「简化版」，不是「完整版」**

不要试图造一个完整的数据库。先造一个只能处理 `SET` 和 `GET` 的 KV 存储。不要试图造一个完整的 Web 框架，先造一个只能处理路由和中间件的 HTTP 服务器。

**第二原则：解决真实痛点**

最好的轮子是「你自己真的会用的」。如果你自己都不想用它，说明你对问题的理解还不够深。

**第三原则：写完后和成熟方案对比**

这是最容易被忽略、却最有价值的步骤。把你的实现和 Redis / Nginx / Go stdlib 做对比：

- 你的数据结构选择和它们一样吗？为什么不一样？
- 你的错误处理和它们差距多大？
- 你的并发模型有它们优雅吗？

**第四原则：把经验写成文章**

参见 [[写作与分享的杠杆]]——把你的造轮子过程写成系列文章，既是对自己的总结，也是对社区的贡献。

### 推荐的造轮子项目（按方向分类）

**系统编程方向**：
- 写一个简化版的 `malloc`（理解内存分配）
- 写一个简化版的 `grep`（理解文本处理和正则引擎）
- 写一个简化版的 shell（理解进程管理和系统调用）

**网络方向**：
- 写一个 HTTP 服务器（从 socket 开始）
- 写一个 RPC 框架（理解序列化和网络通信）
- 写一个负载均衡器

**存储方向**：
- 写一个 KV 存储（基于 LSM-Tree 或 B+Tree）
- 写一个简化版的 Redis（支持 5 个基本命令）
- 写一个日志收集和分析系统

**编译器和语言方向**：
- 写一个 JSON 解析器
- 写一个计算器（从词法分析到求值）
- 写一个 Lisp 解释器

## 「读 -> 造 -> 写」循环：终极学习飞轮

最高效的技术成长模式是把三条路串成一个飞轮：

```
读源码 -> 理解设计 -> 造轮子验证理解 -> 写文章总结 -> 发现新问题 -> 再读源码
```

### 一个具体的例子

1. **读**：阅读 Redis 的 `dict.c`，理解渐进式 rehash 的设计
2. **造**：自己写一个支持 `SET`/`GET` 的 KV 存储，实现渐进式 rehash
3. **写**：写一篇文章《我从 Redis 源码中学到的哈希表设计》
4. **发现**：在写文章时发现自己对 `zipmap` 的理解不清楚 -> 回去再读
5. **循环**

[[John Carmack]] 的整个职业生涯就是这个飞轮的极致体现：他不断阅读图形学论文（读）、实现引擎（造）、写技术文章和演讲（写），循环了几十年。

## 反模式：常见的坑

### 1. 只读不造

- **表现**：读了很多源码，做了很多笔记，但从不动手写
- **危害**：停留在「我知道」的层面，缺乏真正的工程理解
- **解药**：每读完一个重要模块，至少写一个 100 行的简化版实现

### 2. 只造不读

- **表现**：疯狂造轮子，但从不看别人怎么做的
- **危害**：重复发明低质量的轮子，错过学习优秀设计的机会
- **解药**：造轮子之前先读至少 2 个同类项目的源码

### 3. 从头到尾逐行读

- **表现**：打开源码从 `main()` 开始一行一行往下读
- **危害**：很快迷失在细节中，失去全局视角，然后放弃
- **解药**：先用 Top-Down 或 Problem-Driven 方法，建立全局理解后再深入细节

### 4. 造了不写

- **表现**：造了很多轮子，但从不写文章或做分享
- **危害**：知识没有被结构化，很容易遗忘；也没有机会通过反馈发现盲区
- **解药**：每个造轮子项目必须配一篇技术文章（参见 [[写作与分享的杠杆]]）

### 5. 追求大而全

- **表现**：一上来就要「写一个完整的操作系统」或「重写 Kubernetes」
- **危害**：项目太大太复杂，几周后就烂尾
- **解药**：严格遵循「先做最小可用版本」的原则，控制在 2-4 周内能完成

## 大拿们的实践

### [[John Carmack]]

- 几十年持续阅读学术论文、理解最新算法，然后亲手实现
- 每一代 Doom/Quake 引擎都是「读 -> 造」循环的产物
- 他的技术文章和演讲以极度深入著称，因为他真的把每个细节都搞懂了

### [[Rob Pike]]

- Go 语言的设计者之一，同时也是 Unix 早期核心成员
- 他的编程风格以极简著称——这来自几十年阅读和编写系统级代码的积累
- 他的文章《The Practice of Programming》本身就是一本「如何写出好代码」的源码阅读指南

### [[章亦春]]

- OpenResty 作者，深度阅读了 Nginx 和 LuaJIT 源码
- 在此基础上构建了 OpenResty，把 Nginx 变成了一个可编程的 Web 平台
- 他对性能优化的极致追求，直接来自对源码级别细节的理解
